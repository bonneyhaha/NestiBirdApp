WHY NESTI USES MORE RAM THAN A "SIMPLE APP"
===========================================

You said: "a simple app takes <15 MB — this takes over 60 MB."
That comparison is not apples-to-apples. Here is why, and what we did about it.


──────────────────────────────────────────
1. THE WPF FRAMEWORK BASELINE
──────────────────────────────────────────

A "simple app" at <15 MB is typically a console program, a Python/Node script, or a plain
C-language utility. It starts with almost nothing loaded.

A WPF app is different. The moment it launches, Windows loads:

  • CLR / .NET 8 runtime         ~20–30 MB (JIT, GC, thread pool, base class library)
  • WPF presentation layer       ~15–20 MB (layout engine, renderer, input system)
  • DirectX / D3D9/D3D11 layer   ~5–10 MB  (even in SoftwareOnly render mode)
  • WPF resource dictionaries    ~2–5 MB   (brushes, templates, themes)

TOTAL BASELINE: ~40–65 MB before a single line of your code runs.

This is unavoidable for any WPF application. A blank "Hello World" WPF window will show
~50 MB in Task Manager. It is the cost of the framework, not a bug.


──────────────────────────────────────────
2. WHAT WAS LEAKING BEFORE (THE OLD CODE)
──────────────────────────────────────────

The original code used the WpfAnimatedGif library. That library works by calling
ImageBehavior.SetAnimatedSource(image, gifSource) on each Image element.

Internally, WpfAnimatedGif creates a NEW GIF decoder per Image element. That decoder
pre-decodes ALL frames of the GIF into memory (BitmapCacheOption.OnLoad).

For a GIF with 60 frames at 150×150 pixels in BGRA32 format:
  60 frames × 150 × 150 × 4 bytes = 5.4 MB per Image element

With 5 simultaneous notification cards on screen:
  5 × 5.4 MB = 27 MB   (just for the notification avatar GIFs)

The bird GIF added another ~5.4 MB.
All of this was on TOP of the 40–65 MB WPF baseline.

Additionally, WpfAnimatedGif's cleanup was lazy — it waited for the GC, so dismissed
cards left their decoded frames in memory for seconds or minutes. This caused the
"RAM keeps going up" behaviour the user observed.


──────────────────────────────────────────
3. WHAT THE CURRENT CODE DOES
──────────────────────────────────────────

SharedGifPlayer replaces WpfAnimatedGif with a compositing engine that stores:

  ┌─────────────────────────────────────────────────────┐
  │ Per GIF file (not per Image control):               │
  │  • MemoryStream   : compressed GIF bytes  ~50–500 KB│
  │  • _canvas[]      : 1 frame of BGRA pixels   ~4 KB  │  ← for jarvis.gif  32×32
  │  • SharedBitmap   : 1 WriteableBitmap         ~4 KB  │  ← same pixels, shared
  │  • _infos[]       : int metadata per frame  <5 KB   │
  └─────────────────────────────────────────────────────┘

  vs WpfAnimatedGif's:
  ┌─────────────────────────────────────────────────────┐
  │ Per Image element (one per notification card):      │
  │  • All decoded frames in RAM    × N cards = N × MB  │
  └─────────────────────────────────────────────────────┘

All notification cards share the SAME WriteableBitmap. When the frame timer ticks,
one pixel blit updates every card simultaneously — zero per-card overhead.

BitmapCacheOption.None means the decoder does NOT pre-cache all frames. Each frame
is decoded from the MemoryStream only when the timer fires, then immediately released
back to the ArrayPool.

GIF compositing is done correctly:
  • Sub-frame rectangles (Left/Top offsets per frame)
  • Binary transparency  (alpha=0 → keep canvas pixel)
  • Disposal method 2   (clear area to transparent before next frame)
  • Disposal method 3   (restore canvas to previous state)


──────────────────────────────────────────
4. EXPECTED RAM NUMBERS
──────────────────────────────────────────

                          BEFORE          AFTER
  WPF baseline            ~50 MB          ~50 MB   (fixed, unavoidable)
  GIF memory (5 cards)    ~27 MB          ~0.1 MB  (fixed, not per-card)
  GIF memory (bird)       ~5 MB           ~0.1 MB
  Per-card on arrival     +5.4 MB each    ~0 MB extra
  Growth over time        YES (leak)      NO

  Steady-state RAM:       ~82 MB+         ~50 MB

The ~50 MB that remains is the WPF runtime. It cannot be reduced without switching
to a different UI framework (e.g. Win32 C++, Qt, Tauri/WebView2, Flutter).


──────────────────────────────────────────
5. OTHER MEMORY LEAK SOURCES VERIFIED & CLEARED
──────────────────────────────────────────

  ✓ DispatcherTimer per notification  — stopped before removal, Dispatcher uses weak
                                         ref for stopped timers, so they are GC-eligible.
  ✓ Lambda captures in Tick / events  — self-contained cycles, collected by .NET GC
                                         once the notification is removed from _active[].
  ✓ WPF animations (DoubleAnimation)  — AnimationClock released when element is removed
                                         from the visual tree (NotificationsPanel.Children).
  ✓ SharedGifPlayer.FrameChanged subs — ELIMINATED in the new design. WriteableBitmap
                                         auto-notifies, no event subscription per card.
  ✓ AvatarImage.Source               — set to null in Cleanup() so the Image control
                                         releases its reference to SharedBitmap.


──────────────────────────────────────────
6. SUMMARY
──────────────────────────────────────────

  "Simple app <15 MB" uses almost no framework.
  WPF always costs ~40–65 MB baseline — that is the price of its rendering engine.

  The GIF leak we fixed was costing an extra 5–30 MB that GREW with each notification.
  That growth is now eliminated. Memory is stable after startup.

  If a lower baseline is required in future, the UI framework would need to change
  (e.g. WinUI 3, Win32, or a web-based shell). That is out of scope for this fix.
